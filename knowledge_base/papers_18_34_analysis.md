## Paper 18: "Beyond Functional Correctness: Exploring Hallucinations in LLM-Generated Code"
**Authors:** Fang Liu, Yang Liu, Lin Shi, Zhen Yang, Li Zhang, et al. (Beihang University + Huawei)
**arXiv:** 2404.00971v3 (Jan 2026)
**Venue:** Major SE venue

### Key Contribution
Established the **most comprehensive taxonomy of code hallucination** to date.

### Taxonomy (3 Primary + 12 Specific Types)
1. **Requirement Conflicting Hallucinations** (Most prevalent across ALL LLMs)
   - The generated code contradicts the user's stated requirements
   - Subcategories: wrong I/O format, wrong algorithm choice, misunderstood constraints
2. **Code Inconsistency Hallucinations**
   - The generated code is internally self-contradictory
   - Subcategories: variable reuse conflicts, logic branch contradictions
3. **Knowledge Conflicting Hallucinations**
   - The code conflicts with real-world facts (APIs, libraries, math)
   - Subcategories: hallucinated API names, wrong mathematical operations

### Critical Findings
- **3,120 code samples** analyzed across multiple LLMs on HumanEval and MBPP
- **Requirement Conflicting is THE dominant hallucination** — most common across ALL models
- Both **model capability** and **prompt quality** contribute to hallucination
- **Incorrect functionality** is the most common impact, but **readability** and **efficiency** are also degraded

### Mitigation Results
Tested 3 training-free prompt enhancement strategies:
- **Requirement refinement prompts** → reduces Requirement Conflicting hallucinations ✓
- **Chain-of-thought reasoning** → reduces Requirement Conflicting hallucinations ✓
- **Domain knowledge supplementation** → reduces Knowledge Conflicting hallucinations ✓

### How This Informs Our Thesis
- Our 6-type taxonomy was partially correct but needs revision:
  - **Merge** our "API Hallucination" and "Import Hallucination" into their "Knowledge Conflicting"
  - **Add** "Requirement Conflicting" as the dominant type (we called this "Specification Hallucination")
  - **Key insight:** Requirement Conflicting > Knowledge Conflicting in frequency. This means our CPV system's invariant extraction from specs is even MORE important than API validation.

## Paper 20: "A Systematic Literature Review of Code Hallucinations in LLMs"
**Authors:** Cuiyun Gao, Guodong Fan, et al. (HIT, SMU, Sun Yat-sen, Monash, Tianjin, Chongqing)
**arXiv:** 2511.00776v1 (Nov 2025)
**Venue:** ACM TOSEM (accepted)

### Key Contribution
**THE most comprehensive survey** — reviewed 60 papers on code hallucination. This is our primary Related Work source.

### Hallucination Causes (from 60 papers)
1. **Data noise** — conflicting patterns in training data
2. **Exposure bias** — teacher forcing during training creates distribution shift at inference
3. **Insufficient semantic grounding** — model lacks connection to executable semantics
4. **Syntax sensitivity** — code has stricter constraints than natural language
5. **Type system violations** — beyond syntax, type errors account for 94% of compilation errors
6. **External library dependence** — model cannot verify API existence at generation time

### Mitigation Strategies Surveyed
Organized into **two streams**:
1. **Gradient-based (Training-time):**
   - Fine-tuning on hallucination-focused preference datasets
   - Code-specific RLHF with execution feedback
2. **Non-gradient (Inference-time):**
   - RAG (retrieval-augmented generation)
   - CoT (chain-of-thought)
   - CoVe (chain-of-verification)
   - Constrained decoding
   - Post-processing/verification tools

### Open Challenges Identified
1. **Hallucination vs. Mistake** — distinguishing confident-but-wrong from genuinely uncertain
2. **Overconfidence** — high model confidence ≠ factual correctness
3. **Generalization vs. Hallucination** — the tradeoff is fundamental
4. **Private codebases** — hallucination is worse where training data is scarce

### How This Informs Our Thesis
- **Structure our Related Work** using this survey's framework
- **Our CPV falls into "Non-gradient/Inference-time" category** — specifically in the "Post-processing/verification" bucket
- **Key gap identified:** No existing paper combines property-based testing with multi-agent code generation + execution feedback in a single framework. This IS our contribution.
- **Cite the "hallucination vs. mistake" challenge** — our CPV addresses this by making the distinction concrete: if code passes property tests, it's not a hallucination (regardless of whether it's the "intended" solution)

## Paper 22: "Reducing Hallucinations in LLM-Generated Code via Semantic Triangulation"
**Authors:** Yihan Dai, Sijie Liang, Haotian Xu, Peichu Xie, Sergey Mechtaev (Peking University)
**arXiv:** 2511.12288v3 (Mar 2026)

### Key Contribution
**Semantic triangulation** — a theory-grounded framework that decorrelates model errors by transforming the original problem into a "dissociative variant."

### How It Works
1. Given a coding task `d`, apply a **dissociative transformation** to get task `d'`
2. `d'` requires a **fundamentally different algorithm** (e.g., if `d` requires a printer, `d'` requires a parser)
3. Sample solutions to both `d` and `d'` independently
4. Check **consistency** between solutions using a formal relation `φ`
5. If they agree → high confidence the solution is correct

### Four Concrete Triangulation Methods
1. **Problem inversion** — swap input and output
2. **Problem decomposition** — break into sub-problems
3. **Solution enumeration** — enumerate all solutions
4. **Postcondition verification** — auto-infer Hoare-style specs and verify

### Results
- **+24% probability of selecting correct program** over baselines (test generation, metamorphic testing, auto-formalized specs)
- **+26% higher F1 score** in selection-or-abstention scenarios
- Evaluated on LiveCodeBench and CodeElo with GPT-4o, DeepSeek-V3, Gemini 2.5 Flash
- **Machine-checked proofs in Lean** for theoretical requirements

### How This Informs Our Thesis
- **VERY strong related work** — must be cited and differentiated
- **Key difference from CPV:** Semantic triangulation checks **consistency across problem transformations**, while CPV checks **consistency against structural invariants (properties)**
- **Complementary approaches:** Triangulation catches correlated errors; CPV catches logic/boundary errors via fuzzing
- **Their weakness:** Requires the problem to have a meaningful "inverse" or decomposition — doesn't work for all problem types. CPV works for ANY problem with definable invariants.

## Paper 24: "Type-Constrained Code Generation with Language Models"
**Authors:** Niels Mündler, Jingxuan He, et al. (ETH Zurich + UC Berkeley)
**arXiv:** 2504.09246v2 (May 2025)

### Key Contribution
**Type-constrained decoding** — uses programming language type systems to guide code generation during inference.

### Technical Approach
1. Develop novel **prefix automata** for TypeScript
2. Implement **type inhabitation search** — determines if a partial expression can be completed to a well-typed program
3. At each token generation step, **reject tokens that would violate type constraints**
4. Sound approach: guarantees well-typedness of generated code

### Results
- **Reduces compilation errors by >50%**
- **94% of compilation errors are type errors** (not syntax errors!) — this is the key insight
- Increases functional correctness by **3.5-5.5%** for synthesis and translation tasks
- **+37% improvement** in functionally correct repair of non-compiling code
- Evaluated on HumanEval and MBPP TypeScript versions across 2B-34B parameter models

### How This Informs Our Thesis
- **Orthogonal to CPV:** Type-constrained decoding prevents Type 1 (API) hallucinations at generation time; CPV catches Type 2 (Logic) and Type 3 (Boundary) hallucinations post-generation
- **Key statistic for our thesis:** "94% of compilation errors are type errors" → we can cite this to argue that syntax-level fixes are insufficient, motivating deeper property-based verification
- **Future work direction:** Combine type-constrained decoding (during generation) with CPV (post-generation) for a full-stack approach

## Paper 25: "Teaching Large Language Models to Self-Debug"
**Authors:** Xinyun Chen, Maxwell Lin, Nathanael Schärli, Denny Zhou (Google DeepMind + UC Berkeley)
**arXiv:** 2304.05128v2 (Oct 2023)

### Key Contribution
**SELF-DEBUGGING** — teaches LLMs to debug their own predicted code via few-shot demonstrations, including "rubber duck debugging" (explaining code line-by-line).

### Framework (3 Steps per Debug Turn)
1. **Generation:** Model generates candidate programs
2. **Explanation:** Model explains the code in natural language (rubber duck debugging) OR traces execution
3. **Feedback:** Determine correctness (via unit tests or model self-assessment)

### Feedback Types Tested
- **Simple feedback:** Just "correct" or "wrong"
- **Unit test feedback (UT):** Include execution results and failed test details
- **Code explanation feedback (Expl):** Model explains code line-by-line
- **Execution trace feedback (Trace):** Model traces execution step-by-step

### Results
- Spider (text-to-SQL, no unit tests): +2-3% with code explanation, +9% on hardest problems
- TransCoder + MBPP (with unit tests): +12% accuracy improvement
- **Sample efficiency:** matches or outperforms models that generate >10× more candidates
- **Key finding:** Even without unit tests, code explanation alone helps debugging

### How This Informs Our Thesis
- **Foundation for our CPV feedback loop** — SELF-DEBUGGING proves that LLMs can use execution feedback to improve
- **Our CPV goes beyond:** Instead of unit tests (which may be wrong), we use property-based invariants (which are specification-level)
- **Must cite this paper** — it's the foundational self-repair paper
- **Key limitation we address:** Self-debugging still requires either unit tests or self-generated explanations (both can be wrong). CPV uses externally-verifiable invariants.

## Paper 26: "Is Self-Repair a Silver Bullet for Code Generation?"
**Authors:** Theo Olausson, Jeevana Priya Inala, Chenglong Wang, Jianfeng Gao, Armando Solar-Lezama (MIT CSAIL + Microsoft Research)
**arXiv:** 2306.09896v5 (Feb 2024)
**Venue:** ICLR 2024

### Key Contribution
**Critical analysis of self-repair** — shows it's NOT a silver bullet and is bottlenecked by feedback quality.

### Key Findings
1. **Self-repair gains are often modest** when compute cost is accounted for
2. **Better to sample more initial programs** than to do extensive repair
   - 10 initial + 1 repair each > 2 initial + 10 repairs each
3. **Feedback quality is the bottleneck**, not repair ability
   - Using GPT-4 feedback for GPT-3.5 repair → significantly better than self-feedback
   - Using human feedback for GPT-4 repair → +58% repair success rate (33.3% → 52.6%)
4. **Diminishing returns after 2-3 repair rounds** — validates our "debugging decay" insight

### The Repair Tree Framework
```
Specification ψ
    → Initial programs p₁, p₂, ..., pₙ
        → Feedback f₁₁, f₁₂, ... for each
            → Repairs r₁₁₁, r₁₁₂, ... for each
```

### How This Informs Our Thesis
- **STRONGEST validation of CPV's design decisions:**
  1. CPV uses **separate Agent A (code) and Agent B (invariants)** = using a "stronger model" for feedback
  2. CPV limits repair to **2 rounds max** = aligned with their finding of diminishing returns
  3. CPV provides **execution-grounded feedback** (concrete counterexamples from Hypothesis) = higher quality than self-generated natural language feedback
- **Must cite this paper prominently** — it directly motivates why CPV's approach to feedback (property violations with concrete counterexamples) is superior to self-generated explanations
- **Key quote to use:** "Self-repair is bottlenecked by the model's ability to provide feedback on its own code"

## Paper 30: "Constrained Paraphrase Consistency for LLM Hallucination Detection (CCHD)"
**Authors:** Shanshan Lin, Dongsheng Hong, et al. (Fuzhou University + HIT)
**arXiv:** 2606.08158v1 (Jun 2026)

### Key Contribution
Uses **paraphrase consistency** to detect hallucinations — if a model gives different answers to the same question phrased differently, it's hallucinating.

### Technical Approach
- Formulates training as a **constrained optimization problem**
- Two constraint families:
  1. **Paraphrase-consistency:** bound divergence between predictions on original vs. paraphrased inputs
  2. **Label-preservation:** tie paraphrased predictions to ground truth
- Solved via **gradient descent-ascent** over model parameters and Lagrange multipliers
- No inference-time overhead

### Results
- Outperforms FactCG, MiniCheck, AlignScore across 11 factuality tasks
- Macro-F1: 79.73 (best) vs 78.47 (previous best)
- Works with both DeBERTa and Flan-T5 backbones

### How This Informs Our Thesis
- **Different domain** (text hallucination, not code), but the **consistency principle** is powerful
- **Relates to our Differential Testing approach** — if rephrasing the problem produces a different solution, the model is hallucinating
- **Potential enhancement for CPV:** After generating code, rephrase the problem description and regenerate — if solutions differ semantically, flag as potential hallucination

## Paper 31: "Generation Constraint Scaling Can Mitigate Hallucination"
**Authors:** Georgios Kollias, Payel Das, Subhajit Chaudhury (IBM Research)
**arXiv:** 2407.16908v1 (Jul 2024)

### Key Contribution
A **geometry-inspired** approach for mitigating hallucination in memory-augmented LLMs by scaling the readout vector.

### Technical Approach
- Uses **Larimar**, a memory-augmented LLM decoder
- Discovery: readout vectors from memory are **well-aligned directionally** with write vectors but **3-4x shorter in magnitude**
- Simply **scaling readout vectors by 3-4x** significantly reduces hallucination
- Training-free, adds no inference overhead

### Results
- RougeL: 0.72 (scaled) vs 0.39 (baseline) vs 0.49 (GRACE editing method)
- **46.9% improvement** over GRACE, a SOTA LLM editing method
- 1-2 orders of magnitude faster than GRACE

### How This Informs Our Thesis
- **Fundamentally different approach** — works at the inference geometry level, not post-hoc verification
- **Future work direction:** Could memory-augmented code generation with scaled readout reduce code hallucination at the source?
- **Not directly applicable to CPV** but valuable as a "complementary approach" in our Related Work

## Paper 34: "PGS: Effective LLM Code Refinement via Property-Oriented and Structurally Minimal Feedback"
**Authors:** Lehan He, Zeren Chen, Xiang Gao, Zhe Zhang, Lu Sheng (Beihang University, Shanghai AI Lab)
**arXiv:** 2506.18315v2 (May 2026)

### Key Contribution
Introduces the **Property-Generated Solver (PGS)**, a multi-agent framework that uses **Property-Oriented** and **Structurally Minimal** feedback to refine LLM-generated code. This is exactly what we were looking for (previously marked as missing Paper 28)!

### Core Principles
1. **Property-Oriented Feedback:** Shifts from simple I/O mismatch feedback to checking high-level program properties. This prevents the "cycle of self-deception" where models repeat the same logical biases in both code and test generation.
2. **Structurally Minimal Feedback:** Reduces cognitive load by providing the *simplest failing counterexample* (defined empirically as the one with the minimal input token count).

### Framework (Generator & Tester Agents)
- **Generator:** Produces initial code and refines it based on feedback.
- **Tester:** Defines high-level properties from the spec, translates them into executable checks, generates diverse probing inputs, and selects the structurally minimal counterexample.
- **Asymmetry of Verification:** Verifying correctness (writing properties) is inherently easier than generating the solution. PGS exploits this to provide robust guidance.
- **Latent Bug Surfacing:** Property checks injected into code transform silent "Wrong Answer" failures into explicit, machine-checkable "AssertionError"s.

### Results
- Achieves State-of-the-Art (SOTA) on HumanEval, MBPP, LiveCodeBench, CodeContests, and SWE-bench.
- Outperforms heavy debugging frameworks (e.g., MGDebugger, LDB).
- Token count minimization is proven to be the best proxy for feedback simplicity.

### How This Informs Our Thesis
- **Our Closest Competitor & Validation:** PGS completely validates our CPV framework's direction. CPV and PGS share the same core DNA (property-based verification, multi-agent generator/tester).
- **Differentiation:** 
  - PGS relies on an LLM to synthesize inputs (`{IP}`) for property probing. CPV uses **Hypothesis** (property-based fuzzing) which allows for massive, systematic boundary exploration (100s of inputs) rather than relying on LLM-synthesized inputs which might still have blind spots.
  - PGS's "Structurally Minimal" insight (minimal token count) is brilliant and should be integrated into CPV's feedback mechanism (Layer 5).
