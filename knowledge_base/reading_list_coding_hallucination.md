## Category 1: Taxonomy & Benchmarks (What IS Code Hallucination?)

### Paper 18: CodeHalu — Code Hallucinations in LLMs
- **arXiv:** https://arxiv.org/abs/2404.00971
- **Year:** 2024
- **Why read it:** Introduces a **4-type taxonomy of code hallucination** (mapping, naming, resource, logic) and the **CodeHaluEval benchmark** (8,883 samples from 699 tasks). This is the most rigorous hallucination-specific benchmark that exists. Essential for your thesis evaluation methodology.

### Paper 19: CodeMirage — Hallucinations in Code Generated by LLMs
- **arXiv:** https://arxiv.org/abs/2408.08333
- **Year:** 2024
- **Why read it:** Creates a **1,137-sample dataset** of hallucinated Python code from GPT-3.5, categorized by hallucination type against HumanEval and MBPP. Explores detection methodologies using CodeLLaMA, GPT-3.5, and GPT-4. Directly provides the detection baseline you need for comparison.

### Paper 20: A Systematic Literature Review of Code Hallucinations in LLMs
- **arXiv:** https://arxiv.org/abs/2511.00776
- **Year:** 2025
- **Why read it:** The **most comprehensive survey** of code hallucination to date. Covers causes (data noise, exposure bias), code-specific challenges (syntax sensitivity, type systems), mitigation strategies, and evaluation benchmarks. This paper alone could structure your Related Work chapter.

### Paper 21: Beyond Functional Correctness — Exploring Hallucinations in LLM-Generated Code
- **arXiv:** https://arxiv.org/abs/2503.07662
- **Year:** 2025/2026
- **Why read it:** Conducts a **thematic analysis** categorizing hallucinations into 3 primary and 12 specific categories. Goes beyond "does the code work?" to ask "does the code hallucinate even when it passes tests?" — a nuanced question most papers ignore.

## Category 2: Detection & Verification Methods

### Paper 22: Reducing Hallucinations in LLM-Generated Code via Semantic Triangulation
- **arXiv:** https://arxiv.org/abs/2511.12288
- **Year:** 2025
- **Why read it:** Proposes **semantic triangulation** — checking whether solutions remain consistent under non-semantics-preserving problem transformations. A novel approach to decorrelating model errors. This is the closest existing work to our FLARE "self-trace" concept.

### Paper 23: Deterministic AST-Based Hallucination Detection and Auto-Correction
- **arXiv:** https://arxiv.org/abs/2407.08500
- **Year:** 2024
- **Why read it:** Parses generated code into an **Abstract Syntax Tree (AST)** and validates it against a dynamically generated Knowledge Base (built via library introspection). Auto-corrects "Knowledge Conflicting Hallucinations" — exactly our Type 1 (API Hallucination). Critical for your CPV Layer 1 (static analysis).

### Paper 24: Type-Constrained Code Generation with Language Models
- **arXiv:** https://arxiv.org/abs/2504.09246
- **Year:** 2025
- **Why read it:** Introduces **type-constrained decoding using prefix automata** to ensure well-typedness during inference. Demonstrates significant reductions in compilation errors. This is the technical foundation for AST-Constrained Decoding (Approach B in our mitigation strategies doc).

## Category 3: Self-Repair & Execution Feedback

### Paper 25: Self-Debugging — Teaching LLMs to Debug Their Own Code
- **arXiv:** https://arxiv.org/abs/2304.05128
- **Year:** 2023 (seminal paper, still foundational)
- **Why read it:** The foundational paper on **self-debugging**: the LLM generates code, executes it, and uses the execution results to iteratively fix bugs. Establishes the baseline methodology that all subsequent self-repair work builds upon. You MUST cite this.

### Paper 26: Is Self-Repair a Silver Bullet for Code Generation?
- **arXiv:** https://arxiv.org/abs/2306.09896
- **Year:** 2023/2024
- **Why read it:** A **critical counterpoint** to self-repair enthusiasm. Shows that self-repair effectiveness varies dramatically by model size and error type. Syntactic errors are easily repaired; logical errors remain the primary bottleneck. Most gains are concentrated in the first 2-3 repair rounds — directly validating our "debugging decay" insight from Paper 8.

### Paper 27: LLMLOOP — Self-Debugging with Self-Generated Tests
- **arXiv:** https://arxiv.org/abs/2504.01413
- **Year:** 2025
- **Why read it:** Addresses the **"test bias" problem** — when the LLM generates tests that pass its own flawed logic. Uses multi-loop strategies to refine both code AND tests simultaneously. Directly relevant to CPV's separation of Agent A (code) and Agent B (tests).

## Category 4: Property-Based Testing & Formal Verification

### Paper 28: Property-Generated Solver (PGS) — Using Property-Based Testing for LLM Code Generation
- **arXiv:** https://arxiv.org/abs/2502.15279
- **Year:** 2025
- **Why read it:** **THE most relevant paper for your CPV thesis.** Uses collaborative LLM agents — a "Generator" and a "Tester" — where the Tester defines high-level invariants and properties instead of specific I/O examples. Shows **23.1%–37.3% improvement** in pass@1 over traditional TDD methods. This is almost exactly our CPV architecture.

### Paper 29: PropertyEval — Evaluating LLM-Generated Code with Property-Based Tests
- **arXiv:** https://arxiv.org/abs/2407.14764
- **Year:** 2024
- **Why read it:** Provides **property-based tests for HumanEval** benchmark. Shows that standard pass@k metrics overestimate correctness by missing fundamental logical constraints. Provides the evaluation toolkit you'll need for your thesis experiments.

### Paper 30: Understanding the Characteristics of LLM-Generated Property-Based Tests
- **Link:** Published at AIware Conference 2024/2025
- **Why read it:** Demonstrates that **PBT and EBT are complementary** — PBT catches deep logical errors, EBT catches boundary conditions. Combining them increases detection from ~69% to >80%. Validates the CPV tiered approach (static + dynamic).

## Category 5: Advanced Techniques & Frontiers

### Paper 31: Constrained Paraphrase Consistency for LLM Hallucination Detection
- **arXiv:** https://arxiv.org/abs/2606.08158
- **Year:** 2025/2026
- **Why read it:** Uses **consistency checks** across paraphrased prompts to detect hallucinations. If the model gives a different answer to the same question phrased differently, it's likely hallucinating. Applicable to our "Differential Testing" approach.

### Paper 32: Generation Constraint Scaling Can Mitigate Hallucination
- **arXiv:** https://arxiv.org/abs/2407.16908
- **Year:** 2024
- **Why read it:** A **geometry-inspired approach** that scales the readout vector in memory-augmented decoders to achieve training-free hallucination mitigation. This is a fundamentally different approach — no prompt engineering, no external tools, just modifying the inference geometry. Provides a strong "future work" direction for your thesis.
